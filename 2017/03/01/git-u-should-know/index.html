关于git你该知道的

发表于 2017-03-01 | 分类于 fe

基本操作

git status 查看当前状态
git diff 查看修改
git log 日志
git log –pretty=oneline 更清爽的日志
git reflog 历届版本
git reset –hard 1232123 回退到版本号所示的版本
git reset –hard HEAD~1 回退到上一个版本

撤销修改

场景1：当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout – file。

场景2：当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步，第一步用命令git reset HEAD file，就回到了场景1，第二步按场景1操作。

场景3：已经提交了不合适的修改到版本库时，想要撤销本次提交，参考版本回退一节，不过前提是没有推送到远程库。

删除文件

命令git rm用于删除一个文件。如果一个文件已经被提交到版本库，那么你永远不用担心误删，但是要小心，你只能恢复文件到最新版本，你会丢失最近一次提交后你修改的内容。

分支管理

git branch 查看当前分支
git branch name 创建name分支
git checkout name 切换到name分支
git checkout -b dev 创建dev分支并切换到dev分支
git merge dev 把dev分支的工作成果合并到当前分支上
git branch -d name 删除name分支

多人协作

git remote -v 查看远程库信息
git checkout -b branch-name origin/branch-name 在本地创建和远程分支对应的分支
git push origin branch-name 推送自己的修改
git pull 将远程分支的更新合并到本地分支
git branch –set-upstream branch-name origin/branch-name 关联本地分支和远程分支

协作

不把git的分支玩出翔，你都不好意思跟别人说你用过git。

git的分支模型：

  • 永久性分支
    • master branch
    • develop branch
  • 临时性分支
    • feature branch
    • release branch
    • hotfix branch

永久性分支

存在于整个项目的开始、开发、迭代、终止的过程。
master：从项目一开始便存在，用于存放经过测试、已经完全稳定的代码。在任何时候，master存放的应该是可供用户使用的产品代码。master分支代表着产品每一次迭代的新版本，因此，在所有的分支中，master是最不活跃的。
develop：用于开发的分支，每个开发者自己的仓库上也有master和develop，开发者把功能做好后，代码是存放在自己的develop分支中，当测试完毕后，向管理员发起一个pull request，请求把自己仓库的develop合并到源仓库的develop分支。

master所有代码更新应该源于合并develop的代码。

临时性分支

feature：从develop拉出来的一个分支，是开发人员的主要战场，一般要实现一些产品的新特性或功能时，都需要在这里进行开发，比如说我要开发一个朋友圈的功能，我就会拉一个名叫“feature-moments”的分支，并在其中进行开发。
release：预发布分支，在产品发布之前，通常需要进行最后的调整和测试，这时就可以checkout -b一个预发布分支，记性最后的测试。当产品发布时，release分支就可以删除了。
hotfix：一旦产品出现了bug，就需要拉一个hotfix分支进行修复，比如评论功能出现问题，反手就拉一个hotfix-comment的分支进行fix，等到消灭bug以后，再把该分支合并到master和develop，即可删除。
