你的.gitignore失效了吗？

发表于 2017-03-20 | 分类于 fe

惯例逼叨逼

这个周末跟着部门去团建了，最喜欢的项目其实是在宾馆里打王者农药，因为这件事情说出来都有一丝浓浓的消极怠工的味道。
不知怎么的，以往最乐衷于做的事情——一边听歌一边看车窗外的风景，这次也被我用上车就睡的姿势取代了，也正是我为什么这个点写了一个推送。
由于此次分享的知识点之小巧，我都不好意思转发到朋友圈，如果你看到了这篇推送，我只能真诚的隔着手机屏幕对你说一句，“在项目繁多的公共号列表里，你还能看到我，缘分呐！”

背景

上周在公司政策的庇护下，我开心的划了两天水。在周五晚上临走前，准备看一下这两天项目的进度，顺便merge一下，结果发现同事上传了一些不该上传的文件。
在指出了这个问题之后，我下意识的去修改.gitignore，然后按部就班的commit + push。
看一眼结果，有点懵，这些文件居然还在远程仓库中…
于是下意识的打开stackoverflow找了一下问题的原由。

解决

之所以会失效，是由于.gitignore只能忽略那些原来没有被track的文件。换言之，如果一个文件已经被加入到了本地仓库中（被加入了版本管理），则修改.gitignore是无效的。
这又是为什么呢？
瞬间联想到了在调试页面时，浏览器对脚本文件的缓存会导致浏览器不去请求修改后的脚本，所以造成了修改失效的假象。
所以，问题的根源就在于本地git仓库对已添加到版本管理文件的记录缓存（cache）。
既然找到了问题的原因，那么解决方案也随之浮出水面了： 

$ git rm -r --cached .
$ git add .
$ git commit -m 'something u want to say about this commit'
$ git push origin branch-name

Problem solved.
