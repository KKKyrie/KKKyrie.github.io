关于一次SPA项目的反思

发表于 2017-03-27 | 分类于 fe

git vs svn

  • git 是分布式的
  • 分支的不同
    1. svn 中分支就是版本库中的另一个目录
    2. git 中的分支简单有趣，可以在同一个目录下快速的在几个分支之间切换

gulp vs webpack

其实，gulp和webpack不是一回事

  • gulp和grunt是一类，是工具，能够优化前端工作流，比如自动刷新页面、combo、uglify等。
  • webpack和browserify是一类，当然了还有sea.js / require.js，是js的模块化方案。当然了，从名字都能看出来，这四者也可以分为两个阵营：
    • sea.js / require.js：一种“在线编译”的方案，相当于在页面上首先家在一个CMD / AMD 的解释器，让浏览器知道诸如 define、 exports、 module 都是用来做什么的。
    • webpack / browserify：一种“预编译”方案，不需要在浏览器中提前加载好一个解释器，直接在开发环境下写js，AMD / CMD / ES6 都认识，并且编译成为浏览器可以正常执行的js。

总之，gulp是构建工具，webpack是模块化方案。
但随着webpack傍上了React和Vue的大腿，其生态也越发繁荣，各种loaders和plugins层出不穷，已经可以很好的替代gulp的工作了。
之前在网上看到过一位uber的前端工程师说过一段话，甚是经典：

grunt: 我资历最老
gulp: 我有stream，我快快快

browserify: 我支持common.js，简单好用
webpack: 我快快快

SPA

  • 前后端职责分离，架构清晰：前端交互、后端逻辑处理
  • 前后端单独开发、单独测试
  • 良好的交互体验（局部渲染），避免了不必要的跳转和重复渲染。
  • 缺点：不利于SEO、首次加载速度慢（都有相应的应对策略，暂不做讨论）

为什么选择 Vue

  • React Virtual DOM
  • Angular Data Binding

MVVM 数据绑定

本质：通过数据绑定连接 view 和 model，让数据的变化自动映射为视图的更新
过程：在模版编译的过程中，vue会为每一处需要动态更新的DOM节点创建一个指令对象。每当一个指令对象观测的数据变化时，它便会对所绑定的目标节点执行相对应的DOM操作。基于指令的数据绑定使得具体的DOM操作都被合理的封装在指令定义中，所以我们的业务代码只需要涉及模版和对数据状态的操作即可，开发效率和可维护性大大提升。
借助vue-loader，可以实现vue单文件组件系统。

开发阶段

脱离了实践的理论都是废话，有些东西只有自己搞出来才算是自己的。

开发环境的搭建

技术栈： Vue + Webpack + PHP + Mysql
第一部分是先开发前端，所以随便建了个目录就开搞了，但是一旦到了需要决定协议的阶段，就不得不起一个本地服务器了。
用phpStudy开本地服务器，注意要将目录指到开发目录。

页面重构

  • 从简
  • Bootstrap
  • 仿照微博
  • 分割组件，构建组件树（components tree）
  • Vue 列表渲染

数据库

  • 坑爹的字符编码（中文显示乱码问题，utf-8）
  • 前期考虑不充分，导致后期表结构变更

PHP

  • 字符插入数据库时对单引号的处理
    1. magic_quotes_gpc=on
    2. magic_quotes_gpc=off, addslashes，不需要stripslashes

  • 设置数据库字符集

  • 数据的拼装（设计表结构时留下的小坑）

组件间通信

  • 父组件向子组件传递数据（v-for 列表渲染）
  • 子组件触发父组件的事件（父组件v-on监听，子组件$emit触发）
  • 兄弟组件间的通信（借助共用父组件）
